## 📋 Executive Summary (한 페이지 요약)

### 무엇을 결정했나
DOTBIZ Settlement(정산) 시스템에 대해 **5가지 핵심 정책**을 확정하고 프로토타입에 반영했습니다.

> **DOTBIZ Settlement = 우리 입장의 AR(미수금) 추적기**
> 고객사로 송금하는 일은 없으며, 모든 환불·조정은 다음 invoice의 (-) 라인 또는 별도 Credit Note로만 반영됩니다.

### 5가지 결정 한눈에

| # | 결정 | 핵심 |
|:-:|------|------|
| 1 | Accounting 역할 활성화 | 고객사 회계 직무가 invoice 검증 + 분쟁 제기 + 영수증 업로드 |
| 2 | Dispute 해결 = Tickets | 분쟁 제기는 Settlement, 해결은 Tickets로 라우팅 (EllisOP 처리) |
| 3 | Adjustment는 ELLIS 결정 영역 | 고객사 임의 수정 불가, 출처 4종 명시 필수 |
| 4 | POSTPAY 알림 7단계 | 사전 D-7/3/1/Day + 사후 1d/7d/30d aging |
| 5 | AR Aging + Cash basis | 환율 lock at booking time, 90+ 일은 악성 미수금 |

### 결재 요청 사항

| 항목 | 결재 필요 여부 |
|------|:--:|
| 5가지 정책 방향 | ✅ **확인 결재** (현 상태 OK인지) |

## 2. 핵심 결정 5건 — 사유 + 예상 효과

### 결정 #1 — Accounting 역할 활성화 (조회 + 분쟁 + 영수증)

#### 결정 내용
고객사 Accounting 사용자에게 Settlement 페이지 접근 권한을 부여했습니다.
- 조회: 모든 invoice / billing / AR Aging
- 분쟁 제기 (Disputes 탭)
- 송금 영수증 업로드 (Payment Receipts 탭)
- 단, **invoice 발행 / 입금 확인 / 데이터 수정 권한은 없음** (Master 또는 ELLIS 전담)

#### 사유
DOTBIZ가 고객사의 회계 시스템이 아니라는 점을 명확히 했습니다. Accounting의 진짜 일은:
1. 우리에게 줘야 할 돈을 정확히 파악
2. invoice 검증 (계약 단가와 일치하는지)
3. 분쟁 사유 추적 (왜 차감됐는지)
4. 송금 후 증빙 첨부

#### 예상 효과
- 고객사 내부 회계 효율 ↑
- DOTBIZ 입금 매칭 속도 ↑ (영수증 첨부 시)
- 분쟁 제기가 invoice별로 추적 가능 → 분쟁 처리 SLA 측정 가능

#### 위험 신호
- 없음. 권한 분리 명확.

---

### 결정 #2 — Dispute 해결 = Tickets 단일 워크플로우

#### 결정 내용
- **고객사**: Settlement 페이지에서 분쟁 제기만
- **자동 라우팅**: Invoice Dispute 타입의 ticket 자동 생성
- **EllisOP**: Tickets 페이지에서 검토 + Accept/Reject
- **자동 동기화**: Ticket 종결 시 dispute 상태 자동 업데이트

#### 사유
EllisAdmin이 추후 ELLIS 백오피스(별도 앱)로 분리될 예정. 분쟁 해결을 Settlement 페이지 안에 넣으면 양쪽에 같은 UI를 두 번 만들어야 함. **티켓 시스템에 위임하면 분리가 쉬움**.

#### 예상 효과
- DOTBIZ 측 운영 일원화 (모든 이슈가 Tickets에서 끝남)
- 분쟁 해결 SLA 측정 가능 (ticket SLA 활용)
- ELLIS 백오피스 분리 시 작업량 ↓

#### 위험 신호
- 분쟁 ID와 ticket ID 양방향 sync 실패 시 데이터 정합성 깨질 수 있음 → 트랜잭션 + idempotent 동기화 로직 필수

---

### 결정 #3 — Adjustment는 ELLIS 결정 영역 (read-only)

#### 결정 내용
- 고객사는 billing/invoice 수정 불가 (전 read-only)
- Adjustment 발행 권한 = ELLIS만
- 모든 조정 라인에 **출처 4종** 명시 필수:
  - `DisputeAccepted`: 고객사 분쟁 → 인정 → 차액 환원
  - `EllisInternal`: ELLIS 자체 발견 오류 정정
  - `HotelClaim`: 호텔이 사후 추가 청구
  - `ContractAmendment`: 계약 단가 / 수수료 조정

#### 사유
> "**우리쪽에서 고객사한테 돈이 지급될 일은 없어.**"
> "**고객사는 임의대로 수정 불가, 무조건 이의제기만 가능, 결정권은 우리한테 있음.**"

대표이사 결정에 따라 read-only 정책 확정.

#### 예상 효과
- 분쟁 시 출처 추적 100% 보장 (감사 대응)
- 고객사 임의 변경으로 인한 분쟁 차단
- 회계법인 자문 시 명확한 답변 가능

#### 위험 신호
- 출처 4종으로 모든 조정 케이스를 커버할 수 있는지 → 계약법 자문 후 enum 확장 가능성

---

### 결정 #4 — POSTPAY 알림 7단계 (사전 + 사후)

#### 결정 내용
**사전 4단계** (due 전):
- D-7 (P1) / D-3 (P0) / D-1 (P0) / D-Day (P0 + SMS)

**사후 3단계** (overdue):
- +1일 (P0) / +7일 (P0, 끄기 불가) / +30일 (P0 + SMS, 끄기 불가)

#### 사유
PREPAY는 이미 D-7/D-3/D-1/D-Day 자동 알림 체계가 있는데, POSTPAY는 invoice_due_soon (D-2) 1단계만 있었음. **자금 미수 위험 알림이 부족** → 7단계로 강화.

#### 예상 효과
- 송금 누락 ↓ (고객사가 7일 전부터 4번 알림)
- 30일 연체 시 자동 escalation (legal 통보)
- AR Aging 데이터와 일관성 유지

#### 위험 신호
- 알림 피로도 (alert fatigue) → Notifications #4 결정으로 묶음 처리 적용
- 야간 시간대 SMS → Quiet hours 22:00–08:00 우회 (P0 긴급은 살아남음)

### 결정 #5 — AR Aging + Cash basis + FX Lock

#### 결정 내용 (3가지 동시)

**(A) 환율** — ELLIS에 예약 들어오는 시점 lock-in
- billing line의 `lockedFxRate` 필드 기록
- 정산 시점 환율 변동과 무관

**(B) 회계 인식** — Cash basis (입금 시점에 매출 인식)
- Invoice 발행 시: AR(미수금)
- 매칭 완료 시: 매출 인식

**(C) 부채 분류** — 6 Bucket aging
- Current / 1-30 / 31-60 / 61-90 / 90+ (악성) / Disputed

#### 사유 (대표이사 결정 인용)
> "환율은 엘리스에 예약이 들어오는 시점, 즉 고객이 부킹을 한 시점을 적용하지."
> "회계인식 시점은 회사에 입금이 되고 회계처리 한 시점이고"
> "부채는 회계팀이 부채라 인정하는 시점을 부채 시점이지. 그동안은 계속 미수금 상태이고 기간에 따라 악성미수금도 있을 수 있고 티켓이 해결이 안된 건도 있겠지."

→ 그대로 명세에 반영 + AR Aging 카드 UI 신설.

#### 예상 효과
- 회계법인 감사 대응 명확
- 분쟁 중인 건은 aging과 별도 분류 → 운영 가시성 ↑
- 90+일 자동 분류 → 손상 충당금 시점 자동 트리거 가능

#### 위험 신호
- **FX 환율 변동 위험을 DOTBIZ가 흡수**한다는 의미
  - 1년에 5~10% 환율 변동 가능 → 헤지 정책 별도 필요
  - 헤지 안 하면 연 5억 단위 손실 가능 (가정: 연간 정산 100억 × 5%)
