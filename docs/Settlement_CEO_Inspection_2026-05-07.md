## 1️⃣ 구조 검수 (Settlement Architecture)

### 1-1. 정산 모델 (Billing Type)

```
┌──────────────────────────────────────────────────────────┐
│  POSTPAY (후불)            PREPAY (선불)                 │
│  ──────────                ──────────                    │
│  • 디포짓 등록 필수        • 디포짓 없음                 │
│  • 정산주기 단위 집계      • 예약당 1건 인보이스         │
│  • Net-30 등 입금기한      • TL 데드라인까지 결제        │
│  • Non-refundable 가능     • 결제 없으면 예약 확정 X     │
│  • 신뢰 기반 계약          • 카드/송금 즉시 결제         │
└──────────────────────────────────────────────────────────┘
```

**검수 포인트**:
- ☑️ 두 모델 명확히 분리, 하이브리드 없음
- ☑️ 각 모델의 데이터 흐름 1방향 (수정 불가, 취소 후 재예약만)
- ⓒ 디포짓 종류 6가지 (Credit/Floating/Guarantee Deposit/Insurance/Bank Guarantee/No Deposit) — 너무 많지 않은가? **결재 필요**

### 1-2. 데이터 흐름

```
[예약 확정] → [InvoiceWithMatch 생성]
                ↓
        ┌───────┴────────┐
        │                │
    [POSTPAY]        [PREPAY]
        │                │
   [집계 인보이스]   [예약별 인보이스]
        │                │
   [입금 매칭]       [TL 전 결제]
        │                │
   [Cash 인식]       [Cash 인식]
        ↓                ↓
        └────[AR Aging]────┘
                ↓
            [회수 완료]
```

**검수 포인트**:
- ☑️ 단일 진실 소스: `InvoiceWithMatch` 객체 (id, billingType, customerCompanyId, contractId, ohmyhotelEntityId)
- ☑️ Cash 인식 시점 명확 (입금일)
- ☑️ ohmyhotelEntityId로 다국가 법인 구분 (omh-sg, omh-jp 등)

### 1-3. 권한 분리 (RBAC)

| 역할 | 조회 | 인보이스 발행 | 입금 매칭 | 디포짓 등록 | Dispute 처리 |
|------|------|--------------|----------|------------|-------------|
| Master | ✅ 자사 전체 | ❌ | ❌ | ❌ | ✅ 본인 회사 |
| Accounting | ✅ 자사 전체 | ❌ | ✅ | ❌ | ✅ 본인 회사 |
| EllisOP | ✅ 모든 회사 | ✅ | ✅ | ✅ | ✅ |

**검수 포인트**:
- ☑️ 고객사가 인보이스 발행 불가 (자체 발행 차단)
- ☑️ Cross-tenant 차단 (Master/Accounting은 자사만)
- ☑️ EllisOP만 시스템 차원 변경 가능

## 2️⃣ 룰 검수 (Settlement Rules)

### 2-1. 정산 주기 (POSTPAY)

| 주기 | 설명 | 적용 회사 예시 |
|------|------|---------------|
| Monthly | 월말 마감, 익월 N일 입금 | TravelCo (comp-001) |
| Bi-weekly | 격주 마감 | 일부 중대형 |
| Weekly | 주 마감 | 신뢰도 높은 대형 |

**룰**:
- 정산일로부터 **1~31일 이내 입금** (계약별 협의)
- 미입금 시 자동 알림 → SLA 초과 시 EllisOP 에스컬레이션
- 입금 기한 초과 = AR Aging "Overdue" 카테고리

**검수 포인트**:
- ⓒ Net-30이 표준이지만 일부 대형사에 Net-45 협상 가능 → **상한 결정 필요**
- ☑️ 협상 가능하지만 60일은 절대 초과 금지 (정책 권장)

### 2-2. PREPAY 결제 룰

```
[예약 의도]
    ↓
   TL 도래 여부 ?
    ├─ 미도래 + Free Cancel: 데드라인까지 송금 가능
    └─ Non-refundable OR TL 경과: 즉시 PG 카드결제 → 예약 확정
        ↓
    결제 실패 시 → 예약 미확정 (취소 처리)
```

**룰**:
- TL 미도래 + Free Cancel: 데드라인까지 송금 / 카드결제 선택 가능
- Non-refundable: PG 카드결제만 가능 (송금 불가, 즉시 결제 후 확정)
- TL 경과 후: 무조건 PG 카드결제 (송금 불가)

**검수 포인트**:
- ☑️ 송금/카드 분기 명확
- ☑️ Non-refundable 즉시 결제 = 노쇼 리스크 사전 차단
- ✅ **PG 수수료: DOTBIZ 부담** (2026-05-07 결정) → **마진에 반영하여 가격 책정**
  · 카드 결제는 송금 대비 수수료 비용 발생 → 평균 마진 7% → PG 수수료 흡수 후도 수익성 유지
  · 고객사에는 추가 청구 없음 (가격에 이미 포함)

### 2-3. 환율 룰 (FX Lock)

**원칙**: **예약 발생 시점(`fxLockedAt`)의 환율로 영구 고정**

| 단계 | 통화 | 변환 규칙 |
|------|------|---------|
| 호텔 입찰가 | 호텔 통화 | 원본 보존 |
| 고객 표시가 | 고객사 계약 통화 | 예약 시점 환율 적용 |
| Invoice 발행 | 계약 통화 | FX lock 환율 사용 (변동 X) |
| 입금 통화 | 입금 통화 (계약 통화 외도 가능) | 입금 시점 환율로 환산 후 매칭 |

**검수 포인트**:
- ☑️ FX 분쟁 차단 (고객이 계약 시점 환율로 결제)
- ☑️ DOTBIZ 마진은 환율 변동 영향 받음 (헷징 별도 필요)
- ⓒ 다통화 입금 매칭 시 환차손 회계 처리 — **CFO 협의 필요**

### 2-4. AR Aging 룰

```
Current ──▶ 1-30 ──▶ 31-60 ──▶ 61-90 ──▶ 90+ ──▶ Write-off?
   │           │         │         │         │
   │       알림 1차   알림 2차   에스컬     법무 검토
   ↓                  + EllisOP   레이션
독촉 자동화 (시스템) (수동 협의) (재계약 검토)
```

**룰**:
- Current: 입금 기한 도래 전
- 1-30일: 자동 이메일 reminder
- 31-60일: 두 번째 알림 + EllisOP 알림
- 61-90일: Master에게 직접 협의 요청
- 90+일: 법무 검토 / 신용 한도 동결 / 재계약 검토

**검수 포인트**:
- ☑️ 단계별 자동화 명확
- ⓒ 90+ 이후 정책 (Write-off 결재선) — **결재 필요**
- ⓒ 신용 한도 동결 트리거 임계값 (60일? 90일?)

### 2-5. 디포짓 룰 (POSTPAY 전용)

| 종류 | 형태 | 적용 |
|------|------|------|
| Credit by Company | 회사 신용 한도 | 대형사 |
| Floating Deposit | 운영 자금 (사용 시 차감) | 일반 |
| Guarantee Deposit | 보증금 (잔존) | 일반 |
| Guarantee Insurance | 보증보험 | 중소형 |
| Bank Guarantee | 은행 보증서 | 대형 |
| No Deposit | 무담보 (위탁관리만) | 최상위 신뢰 회사 |

**검수 포인트**:
- ✅ **"No Deposit" 적용은 대표이사 승인** (2026-05-07 결정)
  · 매출/신용 점수에 의한 자동 부여 X
  · 케이스별로 대표이사 결재 후 ELLIS Admin이 등록
  · POLICY_CHANGELOG에 사유 + 결재일 영구 기록
- ☑️ Floating Deposit 잔액 관리 자동화 (예약 시 차감)
- ☑️ 6종 모두 ELLIS Admin 등록 필수

### 2-6. 통화 룰

**현재 운영 통화** (계약 발행 통화 기준):
- USD (기본)
- KRW (한국)
- JPY (일본)
- CNY (중국)
- VND (베트남)
- SGD (싱가포르)

**검수 포인트**:
- ⚠️ 6통화 운영 비용 — 환율 매일 갱신 / 매칭 환차 / 회계 분리
- ✅ **신규 통화 추가는 소스마켓 추가 시** (2026-05-07 결정)
  · 신규 시장(국가) 진출 결재와 통화 추가 결재를 통합
  · 별도 통화 추가만 하지 않음 (시장 없이 통화 X)
  · 통화 추가 = 시장 추가 = 대표이사 결재 사항

### 2-7. 보존 / 발송 룰

| 항목 | 정책 | 비고 |
|------|------|------|
| Invoice 보존 | **7년** | 법정 (전자세금계산서법) |
| Voucher 보존 | 7년 | 동일 기준 |
| Voucher QR | **미사용** | 결정 사항 |
| 언어 | 사용자 i18n locale 자동 | EN/KO/JA/ZH/VI |
| 자동 발송 | **ELLIS 백엔드** | DOTBIZ는 read-only |
| 첨부 형식 | PDF + CSV | 분리 |
| 다운로드 권한 | 로그인 + RBAC | Cross-tenant 차단 |

**검수 포인트**:
- ☑️ 보존 기간 법정 준수
- ☑️ ELLIS 백엔드 자동 발송 = DOTBIZ 책임 최소화
- ☑️ Voucher QR 미사용 = 호텔 측 시스템 의존 X (분쟁 차단)

## 3️⃣ 분쟁 처리 (Dispute Workflow)

```
[고객 이의 제기]
    ↓
[Settlement Dispute 티켓 생성]
    ↓
[Master / Accounting 검토]
    ↓
   인정 여부 ?
    ├─ 인정: Credit Note 발행 / Invoice 수정 (영구 기록)
    └─ 거절: 사유 회신 / SLA 내 종결
    ↓
[ELLIS Admin 모든 분쟁 모니터링]
```

**룰**:
- Dispute 사유는 영구 보존 (status 전환만 / 삭제 X)
- Credit Note는 별도 발행 (원본 Invoice 수정 X)
- SLA: High 4h / Medium 24h / Low 72h

**검수 포인트**:
- ☑️ 원본 불변 원칙 (재무 감사 대응)
- ☑️ Credit Note 발행 = 회계 명확
- ❌ **분쟁 자동 인정 없음** (2026-05-07 결정) — 모든 분쟁은 수동 검토
  · 금액 임계값 기반 자동 처리 X
  · Master / Accounting / EllisOP가 사안별 검토 후 인정/거절 결정
  · 자동화 시 부정 사용 / 정책 우회 리스크 차단

## 5️⃣ 종합 결재 결과 (2026-05-07)

### ✅ 구조 검수 (Architecture) — 결재 완료

PREPAY/POSTPAY 이원화 + Cash basis + FX lock + AR Aging 6 buckets + RBAC 3분리 = **모두 표준에 부합**.

핵심 원칙:

| 항목 | 결정 |
|------|------|
| **PG 수수료** | DOTBIZ 부담 + 마진에 반영 (가격에 흡수) |
