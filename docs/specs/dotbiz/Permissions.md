# Permissions & Roles — Functional Spec

> **Status**: NEW. Role model extended from 2 to 3 roles; booking/notification scopes added.
> **Last updated**: 2026-04-24
> **Related**: [AlertSystem.md](./AlertSystem.md) · [MultiEntity.md](./MultiEntity.md)

---

## 1. Overview

A customer company's sub-accounts are organized by **role** and two **scope** fields that control what data each user sees and which notifications they receive.

The initial configuration is **set by the OhMyHotel Account Manager in ELLIS admin during onboarding** — the customer should not have to configure roles from scratch. After onboarding, the customer's Master users can adjust within the allowed boundaries via Master Account > Sub-accounts.

---

## 2. Role Taxonomy

| Role | Page access | Purpose |
|---|---|---|
| **Master** | All pages including Master Account & Settlement | Full company admin. Invites other users, adjusts permissions, handles settlement. |
| **Accounting** | All pages EXCEPT Master Account | Settlement / invoices / aging / top-ups / voucher export. Opened explicitly by Master (not default for every company). |
| **OP** | All pages EXCEPT Master Account & Settlement | Day-to-day booking operator. Searches hotels, makes and manages bookings, opens tickets. |

Authentication guard `hasRole([…])` is used per page to enforce this.

---

## 3. Booking Scope

Controls which bookings a user can **see** in the Bookings list and detail pages, and which bookings they can edit / cancel / re-book.

| Scope | Who sees what |
|---|---|
| `own` | Only bookings the user created (`bookings.created_by = user.id`) |
| `all` | All bookings for the user's company |

Defaults (ELLIS at onboarding):
- Master → `all`
- Accounting → `all`
- OP → `own` (Master can widen to `all` per OP)

### Why OP default is `own`

Most customer companies treat OPs as independent sales channels — each OP owns their own client list and doesn't need (or want) visibility into other OPs' work. Master can open `all` scope for OP team leads or operations managers on a case-by-case basis.

---

## 4. Notification Scope

Controls which **customer-facing alerts** a user receives (independent of the per-type/per-channel alert rules which remain ELLIS-managed — see AlertSystem.md).

| Scope | Who receives what |
|---|---|
| `own` | Alerts generated by the user's own actions (their bookings, their top-up requests, etc.) |
| `all-company` | Every alert for the company (Master oversight) |
| `accounting-only` | Only settlement-category alerts (invoice_overdue, credit_*, topup_*, partial_payment_detected, credit_note_issued). For Accounting role. |

Defaults (ELLIS at onboarding):
- Master → `all-company`
- Accounting → `accounting-only`
- OP → `own`

### Cross-reference with AlertSystem.md

- **Which alerts exist** and **which channels deliver them** → AlertSystem.md (ELLIS-managed, customers cannot modify)
- **Which users receive each alert** → this file, Notification Scope (Master can tune per user)

The two combine: an alert fires in ELLIS → filtered by the recipient's Notification Scope → delivered on the channels the alert type is configured for.

## 5. Data Model (ELLIS)

```sql
ALTER TABLE users
  ADD COLUMN role              VARCHAR(16)  NOT NULL DEFAULT 'OP',
  ADD COLUMN booking_scope     VARCHAR(16)  NOT NULL DEFAULT 'own',
  ADD COLUMN notification_scope VARCHAR(32) NOT NULL DEFAULT 'own';

-- CHECK constraints on enum values
ALTER TABLE users ADD CONSTRAINT users_role_ck
  CHECK (role IN ('Master', 'Accounting', 'OP'));
ALTER TABLE users ADD CONSTRAINT users_booking_scope_ck
  CHECK (booking_scope IN ('own', 'all'));
ALTER TABLE users ADD CONSTRAINT users_notification_scope_ck
  CHECK (notification_scope IN ('own', 'all-company', 'accounting-only'));
```

Existing sub-accounts created before this migration default to:
- Role based on prior `is_master` flag (true → Master, false → OP)
- `booking_scope = 'own'` for OP, `'all'` for Master
- `notification_scope` per role default table above

## 6. Onboarding Flow (ELLIS-side, NEW)

When OhMyHotel onboards a new customer company, the AM uses the ELLIS Customer Setup screen to specify:

1. **Company-level settings**: billingType, contracts, deposit, credit leverage
2. **Initial Master user** (required — must have at least one)
3. **Initial sub-accounts** (optional, bulk CSV import supported):
   - Each with Name / Email / Department / Role / Booking Scope / Notification Scope
4. **Invitation send** → batch email with credentials

This up-front configuration means the customer's first login already has a working team structure. The Master can then adjust as the organization changes, but is not forced to set everything from scratch.

## 7. Customer-side UI

### Master Account → Sub-accounts tab
- Table shows Role / Booking Scope / Notif. Scope columns
- Each row has Edit action → inline select/dropdowns for Role, Booking Scope, Notification Scope
- Add Sub-account dialog includes all three fields
- Bulk import CSV (future)

### Guardrails
- Master cannot demote themselves (self-demotion protection)
- A company must always have ≥ 1 active Master
- Role changes to/from Master or Accounting require ELLIS-side logging (audit)
- Deactivating the last Master is blocked
